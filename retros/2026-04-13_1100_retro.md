## KPT 회고

### Keep (좋았던 것)

- **예외 처리를 단순히 쓰지 않고 케이스를 분리해 설계**: "검색 자체가 빈 결과"와 "LLM이 근거 없다고 판단한 경우"를 같은 예외로 묶지 않고 원인이 다르다는 걸 인지해 다르게 처리. 코드가 동작하는 것과 설계가 정확한 것을 구분해서 생각한 좋은 사례.

- **`dict.fromkeys()`로 순서 보존 중복 제거**: `set()`을 쓰면 간단하지만 순서가 깨진다는 걸 파악하고, LLM이 먼저 언급한 출처가 더 중요하다는 의미론적 이유로 `dict.fromkeys()`를 직접 선택. 동작하는 코드 이상의 의도가 담긴 구현.

- **범위 방어 로직을 스스로 고민**: LLM이 범위 밖 인덱스를 반환할 수 있다는 엣지 케이스를 직접 인지하고 `if i < len(chunks)` 조건을 추가. 외부 시스템(LLM)을 신뢰하지 않는 방어적 프로그래밍 습관.

- **`has_relevant_content` 필드의 필요성을 논리적으로 판단**: 단순히 `answer` 필드만 추가하는 것과 `has_relevant_content`를 별도 필드로 두는 것의 차이를 클라이언트 관점에서 직접 따져보고 결정. "텍스트를 파싱하지 않고도 근거 없음을 판단할 수 있어야 한다"는 이유를 스스로 도출.

- **파일 수정 후 항상 검증하는 습관**: 구현 완료 후 읽어서 확인하는 루틴으로 ABC 불일치, 중복 import 등 잠재적 버그를 커밋 전에 스스로 잡아냄.

### Problem (나빴던 것)

- **ABC 반환 타입 불일치**: 구현체 시그니처 변경 시 ABC도 함께 수정하지 않아 불일치 발생. 검증 단계에서 발견했지만 처음부터 같이 수정했어야 함.
- **미사용 import 인지 늦음**: 작성 의도와 실제 구현 사이의 간극을 바로 파악하지 못함.

### Try (개선해야 할 것)

- **인터페이스 변경 시 구현체 동시 점검**: ABC 수정 시 구현체 시그니처도 같이 확인하는 습관.
- **커밋 전 import 정리 루틴화**: 파일 완성 후 미사용 import 한 번 훑기.

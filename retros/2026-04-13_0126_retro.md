## KPT 회고

### Keep (좋았던 것)

- **자기 경험 기반 설계 적용**: BM25 + 벡터 3:7 RRF 방식은 사전 프로젝트 경험에서 나온 근거 있는 선택이었다. 기술을 따라가는 것이 아닌 경험으로 검증한 방식을 적용했다는 점에서 높은 신뢰도.
- **이해 후 구현**: `ABC, abstractmethod`, `asyncio` 내부 API 문제 등 모르는 개념이 나올 때마다 구현을 멈추고 질문하여 이해한 뒤 진행했다. 이해 없는 복붙이 없었다.
- **임포트 검증 습관**: `python -c "from app.main import app"` 로 전체 의존성 체인을 빠르게 검증한 접근이 실용적이었다.
- **협업 방식 조기 확립**: agent 모드로 시작 → 사용자 중단 → 직접 타이핑 방식 전환이라는 마찰을 통해 세션 초반에 최적의 협업 패턴을 찾아냈다.

### Problem (나빴던 것)

- **개인 파일 원격 push**: `.gitignore` 등록 전에 `git push`가 먼저 이뤄져 개인 작업 문서가 원격에 올라갔다. gitignore 등록과 히스토리 제거라는 추가 작업이 발생했다.
- **agent 모드 초기 진입**: 유저의 작업 스타일(직접 타이핑)을 먼저 파악하지 않고 agent 모드로 진행하려다 3번 중단됐다. 세션 초반에 협업 방식을 먼저 합의했어야 했다.
- **DI 등록 위치 안내 불명확**: 초기 안내에서 파일 내 정확한 위치 정보 없이 상대적 위치로 설명해 혼란이 발생했다.

### Try (개선해야 할 것)

- **gitignore 선행 등록**: 새 파일/디렉토리 추가 전 `.gitignore` 상태를 먼저 확인하고, 민감/개인 파일은 첫 커밋 전에 ignore 처리한다.
- **협업 방식 세션 첫 교환에서 확인**: 세션 시작 시 "직접 구현할지 / 설명만 받을지" 유저 스타일을 명시적으로 확인한 뒤 진행한다.
- **코드 위치 안내 시 파일:라인 명시**: "아래에 추가" 같은 상대적 표현 대신 `파일명:라인번호` 형식으로 정확한 위치를 처음부터 제공한다.

- **`/usage`로 실제 rate limit 상태 확인**: 에러 메시지만 보고 "진짜 rate limit에 걸린 건지" 아니면 "단순 SDK 버그인지" 구분했다. 세션 사용량이 37%였으므로 실제 제한이 아닌 informational 이벤트였고, 이 판단이 이후 처리 방식을 결정하는 근거가 됐다.

- **Fallback 경로 재사용**: 예외 시점까지 누적된 `raw_parts`를 기존 `_parse_response()` 로직이 처리하도록 했다. `_parse_response`는 이미 JSON 파싱 실패 시 `has_relevant_content=False`를 반환하는 환각 억제 로직이 있어, 빈 응답 엣지 케이스도 안전하게 처리된다. 추가 코드 없이 기존 설계가 받아낸다.

- **수정 후 end-to-end 검증 완료**: TXT 업로드, Markdown 업로드, RAG 질의, L1 캐시 히트, L2 Semantic 캐시 히트(유사 질문 "연차는 어떻게 발생하나요?" vs "연차가 어떻게 생기나요?"), 근거 없는 질문 환각 억제, 문서 삭제 시 Redis 캐시 + `__all__` 역인덱스 일괄 무효화, SSE 스트리밍까지 모두 정상 동작 확인했다.

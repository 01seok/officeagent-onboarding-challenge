## KPT 회고

### Keep (좋았던 것)

- **캐시 저장을 fire-and-forget으로 처리한 판단**: LLM 답변을 이미 만든 시점에서 Redis 저장 완료를 기다리면 사용자 응답이 지연된다. `asyncio.create_task()`로 캐시 저장을 이벤트 루프에 위임하고 즉시 반환하는 구조로 응답 지연 없이 캐싱을 붙였다.

- **임베딩 1회 계산 재사용**: L2 Semantic 캐시 조회와 L2 저장 모두 동일한 임베딩 벡터를 사용한다. L1 미스 직후 한 번만 계산하고 이후 흐름에서 재사용해 불필요한 모델 호출을 제거했다.

- **하이브리드 검색 결함 격리**: `asyncio.gather(return_exceptions=True)`로 BM25, 벡터 검색 중 하나가 실패해도 나머지 결과로 RRF를 계속 진행하도록 수정했다. 인프라 장애가 전체 질의 실패로 번지지 않는다.

- **캐시 무효화 역인덱스 설계**: `doc_cache_index:{doc_id}` Set에 캐시 키를 등록해두고, 문서 삭제 시 파이프라인으로 일괄 삭제하는 구조다. TTL만 쓰면 구현이 단순하지만 최대 24시간 오래된 답변이 반환될 수 있어 정확한 무효화를 선택했다.

- **Claude로 설계/구현하고 Codex로 독립 리뷰**: 구현에 참여한 모델이 바로 검증하면 같은 관점에서 보게 된다. codex:rescue를 별도 리뷰어로 투입해 이중 검증하는 흐름을 썼다. Codex 리뷰 결과도 오탐 여부를 직접 판단해 취사선택했다.

### Problem (나빴던 것)

- **codex 리뷰에서 StreamingResponse import 문제 지적**: `from fastapi import StreamingResponse`가 버전에 따라 top-level export가 아닐 수 있다는 점을 처음부터 인지하지 못했다. 안전한 경로인 `from fastapi.responses import StreamingResponse`를 초기 작성 시점에 썼어야 했다.

### Try (개선해야 할 것)

- **문서 업로드 시 캐시 무효화 추가 검토**: 현재 문서 삭제 시에만 `cache.invalidate()`를 호출한다. 같은 doc_id로 재업로드하는 시나리오는 현재 API 설계상 없지만, 업로드 완료 시점에도 무효화 여부를 명시적으로 판단해두는 것이 낫다.
